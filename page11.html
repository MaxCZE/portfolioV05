Mon portfolio API

Objectif du projet

J'ai crée le portfolio sur lequel vous êtes entrain de naviguer actuellement à l'aide d'un logiciel  de création de sites internet, Mobirise, grâce à ce logiciel, j'ai mis en place un interface web de navigation simple à prendre en main. Cependant, au niveau du programme, du code, il ne s'agit pas de ma création. J'ai donc décider de développer entièrement mon portfolio API back. Ce portfolio représente la partie back de l'application web "mon portfolio" pour des raisons de temps, j'ai n'ai pas pu développer la partie front. J'ai donc décidé d'intégrer swagger à mon API et d'héberger la page afin que vous puissiez manipuler mon API plus facilement tout en ayant d'une vision d'ensemble sur les ressources que j'ai mis à disposition ainsi que les éléments techniques intégrés.

Fonctionnement du projet

Sur mon CV, il y a un deuxième qr-code, en plus de celui qui mène à cette page, qui permet d'accès à la page hébergée de swagger. Sur cette page swagger, vous retrouverez toutes les consignes d'utilisation de mon API, libre à vous, à partir de ces consignes, de la tester et la manipuler comme bon vous semble.

Il y a certains éléments notables que je vais détailler ici:

           - L'API est développée en java spring-boot.
           - Le code source est stocké sur un dépôt privé, il n'est donc pas accessible. L'API fonctionne avec un système de base de données embarqué H2. Ce choix technique me permet d'éviter la création et l'initialisation d'une base de données plu compliquée à mettre en place (comme Postgres). Ici la création et l'initialisation de base est gérée par spring. Ce choix technique implique d'avoir un fichier SQL d'initialisation de données dans les ressources de l'applications, il s'agit de données privées, protégées par un système d'authentification sur la page swagger. Rendre le dépôt de ce projet public, irait à l'encontre de l'objectif de la chaine de sécurité mise en place.
           - La sécurité de l'application repose sur une chaine de filtres JWT et Spring-sécurity. L'accès au ressources est donc possible si vous transmettez un token correcte dans le header de chacune de vos requête.
